業務需求與驗收成果之間往往會有相對應的差異，在各個開發成員的角色上，面對需求也常常會有不同的理解，該如何解決或減少這個問題?

## 不同角度看到的需求
一個需求從客戶端，傳遞至PM，經過彙整交由UX完成User Story，再由RD進行開發，最後由QA進行測試並交付給PM與客戶驗收，這樣的流程常常會發生以下問題 : 
1. 驗收結果不如客戶的預期
2. RD接收到的"需求"與實際的"需求"並不相同
![](https://hackmd.io/_uploads/ByDNMAsOh.png)
> 圖片取自Project Cartoon Ver2

## Why this happened?
這就很像我們小時候會玩的遊戲，大家口中都含著一口水，最排頭的人要把一句話依序傳下去，而到了排尾，往往最後的句子都已不成人形。

在不斷地傳遞資訊的過程，很多的細節資訊總在不經意間遺失或是被扭曲，導致最終結果不如意。

很多在驗收時出現的Bug或是結果行為不如預期往往都是溝通需求時的差異，許多RD的重工都是來自想的跟客戶不同。

## 概念介紹 : TDD, BDD, SBE

不論哪個觀念，最終都希望達成
* Do the right thing (東西做對)
* Do the thing right (品質做好)

### TDD 
Test-Driven Development

實例化規格可視為TDD的起點，~~沒有實例的需求都是流氓需求~~
![](https://hackmd.io/_uploads/B1UU65mtn.png)


期望以最迅速的時間開發出**可行走的骨架**，請使用者看並收取回饋再進行改善。

[紅燈/綠燈/重構]循環(Red/Green/Refactor)
![](https://hackmd.io/_uploads/rk-lpAou2.png)

通常分為五大步驟 :
1. 針對一個功能新增test case
2. 執行測試並讓它出錯(功能尚未開發)
3. 開發一個完成最低限度的程式
4. 執行測試並讓它通過
5. 重構程式碼(可讀性、可維護性、擴充性)

TDD優點 :+1: 
1. 最初就確保測試case的撰寫
2. 撰寫程式的切入點從使用方切入，更容易貼切於使用方的想法

TDD缺點 :-1: 
1. 非技術人員較難理解測試情境，較難參與討論
2. 容易有過於發散或過於聚焦的測向(個人經驗)

### BDD & SBE
Behaviour-Driven Development 與 Specification by Example

* BDD相對於TDD，多了一個步驟，**在寫測試前先寫測試規格書**
* 會利用更接近對話的語意來表達應用功能與測試案例，並且可以被轉換為**可執行的測試規格**

* Specification by Example(SBE) : 需求規格實例化
SBE的重點在於 : **案例要具體，操作細節要抽象**

    * 澄清價值，定義目標及範圍
    
        * **如果不做這個需求會怎麼樣**、**有沒有其他替代方法可以解決這個需求**
        
    * 定義出使用者的操作步驟與流程，有了步驟可以更細的檢視各項目的內容
    
        * **操作能達成目標嗎**、**步驟合理嗎**、**流程可簡化嗎**
        
    * 可表達出業務規則的 : Given、When、Then
    
        * **特殊情況有考慮到嗎**、**不同權限的User情況不同**、**規則包含夠全面嗎**
        

#### BDD的工具 : Cucumber

Cucumber是一款可以支援我們完成BDD的工具，他的三大重點如下 : 
1. Automated Tests (自動化測試)
2. Executable Specifications (可執行的需求)
3. Living Documentation (活文件)
![](https://hackmd.io/_uploads/HkmbzWh_3.png)
> https://cucumber.io/docs/guides/overview/

##### 重要術語
他主要由Specification, Scenario, Step組成，彼此之間的關係如下 : 
* 一個Specification包含多個Scenario
* 一個Scenario有很多個Step要完成 (一個Scenario就是一般說的測試情況)
* Specification是可執行的，利用**Gherkin**的語法來描述

##### 使用情境

* Cucumber能夠驗證軟體功能是否符合Specification，並且能夠生成報告來記錄Scenarios的成功與失敗
* 先完成Specification/Scenario的撰寫，再開始開發Production Code
* 完成Production Code後，Scenario擔任Living Documentation和Automated Tests的角色
* Cucumber在C#中的套件叫做**Specflow**，安裝"Specflow"的Nuget即可使用


#### Gherkin語法
##### 目的 : 
1. 撰寫可執行的Specification並具備語意可讀性
2. 能透過Cucumber進行自動化測試
3. 將系統的實際行為文件化

##### Step Keywords

| Keyword | Semantics | 
| -------- | -------- | 
| Given | 前置條件的設定 |
| When | 發生一個事件 |
| Then | 驗證預期結果 |

#### 使用方法
1. 安裝**Specflow**的Nuget套件
2. 在指定位置新增附檔名為 **.feature** 的檔案
3. 撰寫Scenario的實例化規格
4. 撰寫Scenario內的各Step的執行程式碼
5. 在Visual Studio 跟 Jetbrains Rider內都有支援的plugin可以安裝

## 規格實例化&文件結合測試

### 規格實例化的目的
* 對關鍵需求達成共識
* 盡量用具體的方式描述需求，關鍵的訊息越具體越好
* 把需求寫成實例化的代價小，效果好

### 實際案例

* 客戶需求 : 我想要有EDM的功能
* AC(Acceptance criteria) :

    1. 在頁面上能夠完成新增一筆EDM
    2. 能夠設定EDM的內容
    3. 能夠儲存設定的內容
    4. 能夠刪除已建立的EDM資料

## 結論與個人經驗
* 透過高效溝通，利用SBE一路從需求到自動化驗收，將各項串聯起來
* SBE主要會花費更多時間來進行團隊溝通，但較容易達到需求開發測試的一體化
* 有規格實例化，RD開發時對於實際需求可以考量的更完善
* 針對不同需求，可以有評估cost的基準點
* PM驗收成果時，有相對明確的項目來驗收
* 自動化測試的實現時間長、難度高，也不是甚麼都可以自動化，但長遠來看，SBE要有自動化測試才能達到快速且頻繁的驗證
